# Miners REST API

Miners REST API - учебный backend-проект на Go. Приложение симулирует небольшое угольное предприятие, которым можно управлять через HTTP API: предприятие пассивно добывает уголь, нанимает шахтеров, покупает оборудование, возвращает свое текущее состояние и может быть остановлено в конце запуска. Кроме обычных JSON-ответов, проект отдает Prometheus-метрики и содержит готовую Docker Compose-инфраструктуру с Prometheus и Grafana, чтобы за состоянием приложения можно было наблюдать через dashboard.

По функционалу проект работает как маленькая управляемая симуляция. После запуска предприятие автоматически начинает добывать уголь в фоне. Через API можно смотреть текущий баланс, список активных шахтеров, статистику найма и состояние оборудования. Когда на балансе достаточно угля, можно нанимать шахтеров разных классов: `weak`, `normal` и `strong`. Еще можно покупать оборудование `pickaxe`, `ventilation` и `wagon`, проверять цены, смотреть что уже куплено, а в конце завершить предприятие через shutdown и получить финальный отчет по запуску.

Само приложение внутри контейнера слушает порт `8080`. В текущем `docker-compose.yml` этот порт опубликован наружу как `8081`, потому что локальный `8080` часто уже занят во время разработки. Поэтому с хост-машины API нужно открывать через `http://localhost:8081`, а Prometheus внутри Docker продолжает обращаться к приложению по внутреннему адресу `app:8080`.

Для запуска полного стека сначала запустите Docker Desktop и убедитесь, что Docker Engine работает. Затем из директории проекта выполните:

```bash
docker compose up --build
```

Эта команда соберет Go-приложение, поднимет контейнеры приложения, Prometheus и Grafana, а затем начнет показывать их логи в терминале. Если нужно запустить все в фоне и вернуть управление терминалу, используйте:

```bash
docker compose up --build -d
```

После запуска REST API будет доступен по адресу `http://localhost:8081`, Prometheus - по адресу `http://localhost:9090`, а Grafana - по адресу `http://localhost:3000`. Grafana уже настроена: Prometheus добавлен как data source, а dashboard `Miners REST API` будет подхвачен автоматически. Логин по умолчанию - `admin`, пароль - тоже `admin`. При первом входе Grafana может предложить сменить пароль; это можно сделать или пропустить.

Dashboard настроен в быстром учебном режиме: Prometheus собирает метрики раз в секунду, а Grafana обновляет панели раз в секунду. Поэтому после запроса к API или изменения состояния предприятия графики должны реагировать почти сразу. Если контейнеры уже были запущены до изменения конфигурации, перезапустите стек командой `docker compose down`, а затем снова выполните `docker compose up --build`.

Быстрее всего проверить, что API работает, можно запросом статуса предприятия:

```bash
curl http://localhost:8081/enterprise/status
```

В ответ вернется JSON-снимок текущего состояния: баланс угля, активные шахтеры, статистика найма, купленное оборудование и уведомления по балансу. Пока предприятие работает, баланс постепенно растет за счет пассивной добычи. Чтобы завершить симуляцию предприятия, отправьте:

```bash
curl -X POST http://localhost:8081/enterprise/shutdown
```

После shutdown приложение возвращает финальную длительность запуска и итоговое состояние предприятия. Фоновая добыча после остановки прекращается, поэтому баланс больше не должен продолжать расти.

Если нужны только Go API и локальная проверка без Prometheus и Grafana, проект можно запустить без Docker:

```bash
go run ./cmd/app
```

В таком режиме приложение слушает напрямую `http://localhost:8080`, потому что Docker-маппинга портов уже нет.

Prometheus-метрики приложение отдает на `/metrics`. При запуске через Docker Compose их можно проверить так:

```bash
curl http://localhost:8081/metrics
```

Этот endpoint возвращает не JSON, а текстовый формат Prometheus. Там есть стандартные Go runtime и process-метрики, а также прикладные метрики проекта. `miners_enterprise_balance` показывает текущий баланс угля, `miners_active_total` показывает количество активных шахтеров, `miners_hired_total` хранит статистику найма по классам, `miners_equipment_owned` показывает купленное оборудование, `miners_http_requests_total` считает HTTP-запросы по методу, маршруту и статусу, а `miners_http_request_duration_seconds` показывает длительность обработки HTTP-запросов. Prometheus автоматически собирает эти данные по конфигурации из `monitoring/prometheus.yml`, а Grafana читает их из Prometheus и показывает на dashboard.

Игровая модель намеренно простая. Предприятие начинает с пассивной добычи: один уголь каждые две секунды. Слабый шахтер `weak` стоит дешево, имеет небольшой запас энергии и добывает медленно. Обычный шахтер `normal` стоит дороже, но работает быстрее и приносит больше угля. Сильный шахтер `strong` дорогой, зато добывает больше всего и постепенно увеличивает добычу во время рабочего цикла. Оборудование покупается за накопленный уголь; сейчас доступны `pickaxe`, `ventilation` и `wagon`. Состояние купленного оборудования видно и в API-ответах, и в метрике `miners_equipment_owned`.

Основные HTTP endpoints сгруппированы вокруг шахтеров, оборудования, состояния предприятия и управления приложением. `GET /miners/prices` возвращает параметры классов шахтеров. `POST /miners/hire?class=weak&count=3` нанимает шахтеров выбранного класса; класс может быть `weak`, `normal` или `strong`, а количество должно быть не меньше `1`. `GET /miners/active` возвращает текущих активных шахтеров, а `GET /miners/active?class=weak` фильтрует их по классу.

Для оборудования доступны endpoints, которые показывают цены и текущее состояние покупок. `GET /equipment/prices` возвращает цены оборудования. `POST /equipment/{type}/buy` покупает один предмет оборудования, где `type` может быть `pickaxe`, `ventilation` или `wagon`. `GET /equipment` показывает, какое оборудование уже куплено и что можно купить прямо сейчас при текущем балансе.

Endpoints предприятия отвечают за состояние симуляции и завершение запуска. `GET /enterprise/status` возвращает текущий снимок предприятия. `POST /enterprise/shutdown` останавливает предприятие и возвращает финальный баланс, купленное оборудование, статистику нанятых шахтеров и длительность работы. Также есть `PUT /app/close`, который принудительно завершает процесс приложения после короткого goodbye-ответа.

Обычный ручной сценарий проверки выглядит так: сначала поднимается стек, затем проверяется статус предприятия, после накопления небольшого баланса нанимается шахтер, dashboard в Grafana начинает показывать изменения, а в конце предприятие останавливается:

```bash
curl http://localhost:8081/enterprise/status
curl http://localhost:8081/miners/prices
curl -X POST "http://localhost:8081/miners/hire?class=weak&count=1"
curl http://localhost:8081/enterprise/status
curl -X POST http://localhost:8081/enterprise/shutdown
```

Структура проекта небольшая и разделена по слоям. `cmd/app` содержит точку входа приложения. `internal` хранит доменные типы и конфигурацию, например классы шахтеров, типы оборудования и цены. `logic` содержит жизненный цикл предприятия и бизнес-правила. `myHttp` отвечает за роутинг, HTTP handlers, DTO, middleware и Prometheus-интеграцию. В директории `monitoring` лежат конфиги Prometheus и Grafana provisioning, а `Dockerfile` и `docker-compose.yml` описывают запуск всего стека локально.

Тесты запускаются стандартной командой:

```bash
go test ./...
```
