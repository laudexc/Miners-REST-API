# Miners REST API (учебный проект)

## Что это за проект
Это учебная игра-симулятор "Угольная шахта", в которой управление идет через HTTP API.  
Задача игрока: как можно быстрее купить всё необходимое оборудование, используя стартовый пассивный доход и найм шахтёров разных классов.

## Основная цель
1. Запустить предприятие с базовым доходом 1 уголь в секунду.
2. Грамотно нанимать шахтёров и тратить уголь.
3. Купить всё оборудование:
   - кирки (3000),
   - вентиляция (15000),
   - вагонетки (50000).
4. После покупки всего оборудования завершить игру и получить финальную статистику (включая время прохождения).

## Ключевые игровые правила
1. Уголь является и ресурсом, и валютой.
2. Шахтёры работают параллельно (каждый живёт своей жизнью до конца энергии или остановки игры).
3. Маленький и нормальный шахтёры дают фиксированную добычу, сильный шахтёр прогрессирует.
4. Все активные шахтёры должны корректно завершаться при остановке предприятия.
5. Управление игрой происходит через REST-подобные HTTP-эндпоинты.

## Подсказки по реализации (пошагово, без кода)
1. Сначала опишите доменные сущности:
   - предприятие,
   - шахтёр и его класс,
   - оборудование,
   - статистика сессии.
2. Продумайте единый поток денег:
   - кто увеличивает баланс,
   - кто уменьшает баланс,
   - как избежать гонок данных.
3. Реализуйте жизненный цикл предприятия:
   - запуск,
   - рабочее состояние,
   - корректное завершение.
4. Добавьте фоновые процессы:
   - пассивный доход,
   - работа нанятых шахтёров,
   - остановка по контексту.
5. Введите слой бизнес-операций:
   - найм шахтёра,
   - покупка оборудования,
   - чтение промежуточного состояния,
   - завершение игры с финальным отчётом.
6. После этого проектируйте HTTP-контракт:
   - отдельные маршруты для шахтёров, оборудования и состояния предприятия,
   - понятные статусы ошибок (не хватает угля, уже куплено, игра завершена и т.п.).
7. На каждом шаге проверяйте инварианты:
   - баланс не уходит в минус,
   - нельзя покупать одно и то же оборудование повторно,
   - финал доступен только после выполнения условий победы.
8. Отдельно проверьте graceful shutdown:
   - больше не принимаются новые действия,
   - фоновые процессы останавливаются,
   - финальная статистика формируется один раз.

## Примерная архитектура проекта
Ниже ориентир, который удобно держать в голове (названия директорий можно адаптировать под себя):

1. `cmd/app`  
   Точка входа: запуск HTTP-сервера, инициализация зависимостей, обработка сигнала завершения.

2. `internal/domain`  
   Чистые модели и правила предметной области:
   - типы шахтёров,
   - оборудование,
   - состояние предприятия,
   - структура статистики.

3. `internal/miner`  
   Реализация поведения разных классов шахтёров и их рабочего цикла.

4. `internal/enterprise`  
   Центральная логика игры:
   - баланс,
   - активные шахтёры,
   - покупки,
   - агрегация статистики,
   - управление контекстом остановки.

5. `internal/service`  
   Сценарии уровня приложения (use-case слой):
   - найм,
   - покупка,
   - получение статуса,
   - завершение игры.

6. `internal/transport/http`  
   HTTP-обработчики, роутинг, валидация входных данных, маппинг ошибок в HTTP-ответы.

7. `internal/store` (опционально)  
   Если нужно отделить хранение состояния/счётчиков от бизнес-логики.

8. `internal/config` (опционально)  
   Константы и настройки проекта (цены, параметры шахтёров, адрес сервера).

## Минимальный набор API-групп
1. Шахтёры:
   - просмотр доступных классов и стоимости найма,
   - найм,
   - список активных (в том числе фильтр по классу).
2. Оборудование:
   - список цен,
   - покупка,
   - текущий прогресс покупки.
3. Предприятие:
   - текущее состояние (баланс, активные, история найма),
   - завершение игры и получение итоговой статистики.

## Что важно для успешной сдачи
1. Корректная конкурентная работа (без гонок и потерь данных).
2. Предсказуемые HTTP-ответы и явные ошибки.
3. Чистое завершение приложения: без зависших фоновых задач.
4. Понятная структура проекта, чтобы легко расширять новыми типами шахтёров.
